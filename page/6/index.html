如何使用Git更新博客（一次失败的尝试）

开门见山，先来说一下我用git发布博文的整体结构。

  • 博客服务器一枚
  • Git@OSC帐号一枚
  • 本地电脑一台

发布工作的流程是这样的：

  1. 本地编写完成之后在博文目录提交并push到OSC的仓库
  2. 博客服务器从OSC的仓库中pull得最新博文。

这样博客服务器和本地电脑上面的git都是充当客户端的git，本来想看一看有没有直接上传的办法，找了一会没找到就采取这种办法了，不过这样也好，我的博文有四处存放点（本地+博客服务器+OSC仓库+坚果云）。

为了保证每次打开博客看到的都是最新的情况，我在GitBlog的index.php里面加了一句运行本地shell文件的命令，如下：
exec(".../gitpull.sh");
这样每次打开博客都会自动去远程仓库pull一下，总感觉pull频繁了会有安全隐患呢。。。先不管了，以后再说吧。其实刚才也尝试了crontab的方法（linux的定时器），设定的是每五分钟pull一次，但是好像没什么效果，加上这么做可能不能实时更新，就放弃了。以后的话看看改善一下gitpull这个脚本吧。

对了，我现在用的MarkDown是一个开源的Qt的本地编辑器CuteMarkEd，每次打开都要手动复制博文的前缀也是很麻烦的，所以最近想要在这个CuteMarkEd的基础上添加一些适合GitBlog的功能，这样用起来应该会方便一些吧。

但是Git的方案失败了，因为在网页里面的php代码不能执行git pull，有说是权限问题的，我尝试了几次不行就放弃了，因为对php不熟悉。

但是！我现在也实现了自动发布博文功能！是用Dropbox实现的，稍等我就来记载一下

套接字的使用现在只是实现了特定的功能，容错性什么的暂时没有考虑。

Socket编程的步骤就是连接-》发送消息-》接收消息（出错处理）。
QT对Socket进行了封装，封装为了QTcpSocket类，调用这个类的connectToHost函数即可对目标进行一次连接尝试。（如果连接不上会返回连接错误）
我写了一个自己的MyTcp，在这个类里面有一个QTcpSocket变量。
通过Qt的信号与槽的机制，将消息发送成功与接收消息绑定在一起。

void MyTCP::SendRequest(qint8 slaveaddress, qint8 functioncode, qint16 startaddress, qint16 datanumber)
{
QByteArray line;
QDataStream out(&line, QIODevice::WriteOnly);
out.setVersion(QDataStream::Qt_DefaultCompiledVersion);
out << qint8(00) << qint8(00) << qint8(00) << qint8(00);//protocal identifier 4byte
qint16 messagelen = sizeof(slaveaddress) + sizeof(functioncode) + sizeof(startaddress) + sizeof(datanumber);
out << messagelen;//High 8-bit ModBus & Low 8-bit message length
out << slaveaddress;//slave address
out << functioncode;//function code
out << startaddress;//High 8-bit & Low 8-bit of start address
out << datanumber;//High 8-bit & Low 8-bit of data number
tcpSocket.write(line);

}

qint64 availabelnum = tcpSocket.bytesAvailable();
QByteArray res = tcpSocket.readAll();
}

QtNote:Qt5.6缺少QODBC驱动

前阵子发现Qt5.6终于发布了之后，满心欢喜的安装了Visual Studio 2015和Qt5.6，最近发现Qt5.6默认不带QODBC驱动了，以前的程序连接不上SqlServer了，很是苦恼，于是尝试了以下两种解决办法，其中第一种是不可行的，大家不要尝再去试。

##从Qt5.5中拷贝QODBC（不可行）
秉承着少什么给什么的朴素主义程序员原则，我重新安装了Qt5.5，并且从Qt5.5.0\5.5\msvc2015\plugins\sqldrivers\下面拷贝出了以qsqlodbc开头的三个文件到Qt5.6对应的文件夹中，运行程序，还是显示无法加载QODBC，但是系统已经检测到sqldriver文件夹下面有QODBC了，但是由于版本不同无法使用，所以此路不通。

##从Qt5.6源码中编译QODBC
虽然不知道为什么Qt5.6默认不自带QODBC了，但是还是可以从源码中编译出QODBC的，方法步骤也挺简单的，取自Qt5.6 windows下vs2015编译QODBC作者：蜗牛的信仰。

主要有五步：

  1. 打开Visual Studio提供的CMD（我这里是VS2015 x64 Native Tools Command Prompt）
  2. 进入Qt5.6.0\5.6\msvc2015\bin，执行qtenv2.bat
  3. 进入Qt5.6.0\5.6\Src\qtbase\src\plugins\sqldrivers\odbc，执行qmake odbc.pro
  4. 执行nmake（开始编译）
  5. 编译完成之后进入Qt5.6.0\5.6\Src\qtbase\plugins\sqldrivers中将所有qsqlodbc开头的文件（注意是所有的文件，不只是三个）拷贝到Qt5.6.0\5.6\msvc2015\plugins\sqldrivers

但是当我想要编译32位QODBC的时候（我同时也安装了32位Qt），上面的方法却行不通了，就算我用的是VS2015 x86 Native Tools Command Prompt也不行，说模型是x86的，但是目标机器是x64的，编译失败。

这时候可以打开Visual Studio，使用Qt的菜单栏Open Qt Project File(.pro)打开Qt5.6.0\5.6\Src\qtbase\src\plugins\sqldrivers\odbc中的odbc.pro文件，然后将用Configuration Manager新建一个solution platform为x86的，然后分别Debug build一下和Release build一下。然后去Qt5.6.0\5.6\Src\qtbase\src\plugins\sqldrivers\odbc\Debug|Release里面拷贝东西就好了。

另外，其实Debug build的时候会在Qt5.6.0\5.6\Src\qtbase\plugins\sqldrivers里面生成Debug的库文件，但是Release的时候不会在那个目录下生成，因为新建solution platfrom的时候没有设置生成地点，所以就默认在odbc工程根目录下生成了。
